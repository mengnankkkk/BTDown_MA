# agent.md

## 仓库级强制约束

本文件是整个代码仓库的必读约束。后续所有实现、重构、补充与维护都必须遵守以下规则。

## 1. 总体开发原则

- 架构必须清晰，模块职责必须明确。
- 抽象必须完整，避免把多个层级的职责混写在一个文件或一个模块中。
- 优先保证可维护性、可扩展性和结构清晰度，而不是只追求临时可运行。
- 实现时必须先考虑分层和边界，再开始写具体逻辑。
- 开发顺序固定为：**先后端，后前端**。

## 2. 文件与模块拆分要求

- 单个文件不能过长，必须主动拆分。
- 如果一个文件同时承担多种职责，必须继续拆分为更小的模块。
- 同一文件中不要混合：
  - 数据结构定义
  - 核心业务逻辑
  - 接口适配逻辑
  - 启动与装配逻辑
  - 工具函数堆叠
- 公共能力应下沉为独立模块，不允许在多个文件中重复堆砌相似实现。
- 拆分标准以“职责单一、阅读顺畅、便于定位和替换”为准。

## 3. 后端架构要求

后端代码设计必须参考 **Java 常见企业级分层架构** 的思路来组织，即使实现语言是 Go，也必须遵循清晰的分层与边界。

推荐遵循以下分层思路：

- **入口层 / 接口层**
  - 负责 Wails 绑定入口、HTTP 接口、参数接收、结果返回。
  - 不在这一层堆积核心业务逻辑。

- **应用层**
  - 负责编排用例、调度流程、组织多个领域动作。
  - 这一层描述“做什么”，而不是“底层怎么做”。

- **领域层 / 核心业务层**
  - 负责 session、torrent、stream、scheduler、player 等核心模型与核心规则。
  - 这一层必须保持高内聚，避免被外部框架细节污染。

- **基础设施层**
  - 负责外部播放器启动、文件系统、配置、日志、torrent 依赖封装、网络能力封装等。
  - 不允许把基础设施细节直接散落到业务代码各处。

- **装配层**
  - 负责依赖初始化、组件组装、生命周期管理。
  - 不允许在入口函数中堆满初始化细节。

后端必须做到：

- 明确分层
- 明确依赖方向
- 明确接口边界
- 明确对象职责
- 明确模块归属

禁止事项：

- 把所有逻辑直接塞进 `main.go`、`app.go` 或某个单一 service 文件
- handler 直接操作底层依赖并串联全部业务流程
- 把配置、日志、业务、播放器启动、torrent 控制混在一起
- 为了省事而放弃分层

## 4. 前端架构要求

前端必须做完整、清晰、合理的拆分，不能写成单页面堆砌结构。

前端至少要关注以下边界：

- 页面层
- 业务组件层
- 通用组件层
- 状态管理层
- 服务调用层
- 类型定义层
- 工具层

前端必须做到：

- 页面职责明确
- 组件拆分合理
- 状态来源清晰
- 数据流向清楚
- UI 结构可维护

禁止事项：

- 单个页面文件承载大量逻辑、样式、状态和网络调用
- 多个组件重复保存相同业务状态
- 把所有调用都直接写在页面组件内部
- 组件命名、目录组织和职责边界混乱

## 5. 开发顺序要求

必须严格遵守以下顺序：

1. 先完成后端核心结构和能力边界
2. 再完成后端可运行主链路
3. 再开始前端页面与交互接入
4. 最后做前后端联动体验完善

任何情况下，都不要在后端主链路和接口边界尚未稳定前，过早投入大量前端实现。

## 6. 测试要求

本仓库当前不要求编写测试文件。

执行要求如下：

- 不需要新增单元测试文件、集成测试文件或端到端测试文件。
- 采用直接上手运行、联调、手工操作、真实链路验证的方式进行确认。
- 验证重点放在：
  - 主流程是否可运行
  - 关键交互是否可达成
  - 配置、启动、播放、状态流转是否正常

说明：

- 当前阶段强调快速构建主链路和真实可运行能力。
- 不要因为补测试文件而拖慢主路径实现。

## 7. 实现风格要求

- 优先写结构清楚的代码，不写堆叠式代码。
- 优先写容易替换和扩展的模块，不写强耦合实现。
- 优先写职责清晰的接口，不写含糊不清的万能对象。
- 遇到复杂逻辑时，先拆职责，再写实现。
- 任何明显过大的文件、函数、组件，都应继续拆分。

## 8. 执行口径

后续在本仓库中进行开发时，默认按照以下口径执行：

- 先后端，后前端
- 先分层，后编码
- 先拆分，后堆实现
- 先保证结构清晰，再补功能细节

如果后续需求与本文件冲突，默认优先参考最新明确的人类指令；若无新指令，则始终以本文件为准。
