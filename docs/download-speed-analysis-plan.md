# 下载速度与边下边播提速分析报告

## 1. 文档目的

本文档基于当前仓库代码、`todolist.md` 与现有 `docs` 中的计划文档，对 BT 下载链路和边下边播链路做只读分析，回答三个问题：

1. 当前下载与流式播放链路已经做到什么程度。
2. 下载速度、首播速度、seek 恢复速度是否还有提升空间。
3. 后续应该按什么顺序优化，才能在收益、风险和改动面之间取得更好的平衡。

本文档只做分析和计划，不包含业务代码修改。

关联文档：

- 当前状态快照：`../todolist.md`
- 外部项目调研：`open-source-streaming-performance-research.md`
- 执行路线图：`streaming-execution-roadmap.md`

---

## 2. 当前项目状态概览

结合 `todolist.md` 与现有实现，当前项目大体处于 Phase 2 到 Phase 3 之间：

- 基础下载链路已经打通：`AddMagnet -> Wait GotInfo -> 选主文件 -> Download`
- 本地 HTTP Range 流媒体链路已经打通
- 主文件自动选择已经具备
- `selectedFile.NewReader()` 的边下边播模式已经启用
- 已有首尾预热、动态 readahead、Range 观测等基础能力
- 但真正决定播放体感的请求驱动调度器仍未完整实现

从代码状态看，这不是“完全没有优化能力”，而是“已经有一部分优化手段，但关键时机和系统协同还不够完善”。

---

## 3. 现有下载/流式链路梳理

### 3.1 主链路

当前播放相关的核心路径如下：

1. 会话创建后，由 `torrent_runtime_manager` 启动 magnet 解析和 metadata 获取。
2. metadata 就绪后，自动选择主文件。
3. 对主文件执行：
   - `file.SetPriority(torrent.PiecePriorityHigh)`
   - `file.Download()`
   - `preheatFilePieces(...)`
4. HTTP 层通过 `ServeContent` 暴露本地流地址。
5. 打开流时，通过 `selectedFile.NewReader()` 创建 reader，并启用：
   - `SetResponsive()`
   - `SetReadahead(...)`
6. 每次 Range 请求结束后，记录访问数据，并按 Range 位置更新 priority window。

### 3.2 当前已经具备的优化点

- 文件级高优先级下载
- 文件头尾预热
- Reader 响应式读取
- 可调 readahead
- seek gap 检测
- boost window 相关配置项
- Range 访问耗时、首帧、seek 恢复、buffer hit ratio 等观测字段

说明当前系统并不是“没做优化”，而是“优化手段已部分存在，但没有完全形成闭环”。

### 3.3 当前尚未被显式建模的外部变量

除了代码内部调度，以下变量也会显著影响“下载速度”和“边下边播体感”，但当前文档还没有单独建模：

1. **播放器请求模式**
   - 单段 `Range`
   - 多段 `Range`
   - `bytes=-N` 尾部探测
   - `If-Range` 条件请求
2. **容器与索引布局**
   - MP4 的元数据是否靠前
   - 是否需要尾部索引才能开始稳定播放
3. **资源供给质量**
   - 资源热度
   - tracker / DHT 可发现性
   - 入站连接能力与 NAT 穿透效果
4. **机器本地环境**
   - SSD / HDD
   - 多会话同时运行
   - 后台高频刷盘

这几类变量不会推翻前面的主结论，但会显著影响：

- 首帧耗时的波动区间
- seek 恢复时间的真实上限
- 对同一优化结论的复现稳定性

## 4. 关键瓶颈归因

这里把问题拆成三层：调度时机、持久化分层、设置生效范围。

### 4.1 瓶颈一：Range 提权时机偏晚

#### 现象

在当前实现中，播放器的 Range 请求先进入 `ServeContent`，真正的 Range 记录和 `applyRangePriority(...)` 发生在响应结束之后。

#### 证据

- `internal/controller/stream_controller.go`
  - 先 `OpenStream(...)`
  - 再 `http.ServeContent(...)`
  - 最后才 `runtimeManager.RecordRangeActivity(...)`
- `internal/service/torrent_runtime_manager.go`
  - `RecordRangeActivity(...)` 内部才会判断 seek、更新 readahead、调用 `applyRangePriority(...)`

#### 影响

这会带来一个关键问题：

- 对“下一次请求”可能有帮助
- 但对“当前这次首播请求”或“当前这次 seek 请求”的帮助有限

换句话说，当前策略更像“事后补偿”，不是“请求到来前的主动调度”。

#### 判断

这是当前最值得优先优化的点，因为它直接影响：

- 首帧速度
- seek 恢复速度
- 播放器缓冲等待时间

---

### 4.2 瓶颈二：高频观测与会话持久化尚未清晰分层

#### 现象

当前仓库已经出现了 `internal/repository/impl/file_session_repository.go`，说明会话落盘治理已进入实现面，但当前第一阶段提速不应把它和 Range 热路径改造混在一起。

#### 证据

- `todolist.md` 已将这项工作明确排在请求开始即提权之后
- `docs/streaming-execution-roadmap.md` 中也把该问题放在 Phase D
- 当前最直接影响首播和 seek 的仍然是 Range 提权时机，而不是持久化策略本身

#### 为什么它仍然重要

高频观测、Range 统计和会话状态都会持续写回 session 数据。

如果后续持久化层继续沿着高频全量刷写演进，就会形成：

- 不必要的磁盘写放大
- UI 观测与下载线程共享 I/O 资源
- 多会话下更明显的系统抖动

#### 判断

这是明确需要进入后续优化清单的结构性问题，但不应抢在“请求开始即提权”之前处理。

当前正确顺序是：

1. 先修正 Range 请求热路径
2. 再治理高频观测与持久化分层

---

### 4.3 瓶颈三：设置更新不一定真正作用到底层 torrent client

#### 现象

当前 `DownloadRateLimitKiBps`、`UploadRateLimitKiBps`、监听端口、端口转发等设置，是在创建 `torrent.Client` 时注入的。

设置页更新后，运行期只是调用 `ApplySettings(...)` 替换内存中的 `settings`，没有看到重建 client 或回灌 limiter 的逻辑。

#### 证据

- `internal/service/torrent_runtime_manager.go`
  - `NewTorrentRuntimeManager(...)` 中初始化：
    - `ListenPort`
    - `UploadRateLimiter`
    - `DownloadRateLimiter`
- `internal/controller/settings_controller.go`
  - 更新后调用 `runtimeManager.ApplySettings(updated)`
- `internal/wails/app_bindings.go`
  - 同样只是调用 `ApplySettings(updated)`
- `ApplySettings(...)`
  - 仅更新 manager.settings

#### 影响

这意味着两类风险：

1. 用户修改限速/端口后，可能以为已生效，实际上没有完全生效
2. 排查“为什么速度没变化”时，容易误判为网络问题或种子问题

#### 判断

这不是默认情况下的唯一瓶颈，但会影响：

- 调参有效性
- 故障定位效率
- 真实测速结论可信度

---

### 4.4 边界四：协议与容器语义会放大或掩盖真实瓶颈

#### 现象

当前实现把播放器访问简化为“拿到一个 `Range` 起点，然后围绕这个起点提权”。这对单段线性请求是成立的，但对真实播放器行为并不总是够用。

#### 证据

- Go `ServeContent` 官方文档明确指出，它会正确处理：
  - `Range`
  - `If-Range`
  - 其他条件请求头
- MDN 对 `Range` 的说明指出：
  - 一个 `Range` 头里可以同时请求多个区间
  - 也可能发起尾部 `suffix range`
  - 服务端也可能忽略 `Range`，直接返回 `200`
- 当前实现中：
  - `stream_controller.go` 只在 `ServeContent(...)` 结束后统一记录一次访问
  - `parseRangeStart(...)` 只取首个区间起点
  - `recordRangeActivity(...)` 只围绕一个 `rangeStart` 做 seek 判定与窗口提权
- FFmpeg 文档示例持续强调 `movflags=+faststart`，说明“媒体元数据位置”会直接影响网络/流式消费方式

#### 影响

这会带来几类容易被低估的问题：

1. **多段 Range 被当成单段 Range 处理**
   - 观测数据会失真
   - 提权窗口可能照顾错了真正关键的区间
2. **尾部探测与容器索引读取容易被误判为普通 seek**
   - 可能过度重置窗口
   - 也可能错误降低旧窗口
3. **`If-Range` / `200` 回退会影响首播与恢复统计**
   - 当前指标里未单独区分这类请求
4. **MP4 索引位置会改变播放器的访问序列**
   - 如果元数据不靠前，播放器更可能先探尾部
   - 这会让“头部预热是否足够”出现很大资源差异

#### 判断

这不是当前第一优先级瓶颈，但它决定了：

- 你的观测是否可信
- 你的 seek 判定是否容易误判
- 你的窗口提权是否真正对准播放器最急需的数据

因此它应该进入后续优化文档，而不是继续作为隐含前提存在。

---

## 5. 现有优化能力的边界判断

需要特别说明：当前系统并不是完全没有“请求驱动”的能力。

### 5.1 已有的有效部分

- `Reader.SetResponsive()` 已启用
- `Reader.SetReadahead(...)` 已启用
- Range gap 检测已存在
- `applyRangePriority(...)` 已存在
- `StreamBoostWindowPieces`、`StreamSeekGapFactor`、`StreamDeprioritizeOldWindow` 等参数已存在

### 5.2 还没有形成闭环的部分

- 优先级窗口切换时机不够早
- `SubscribePieceStateChanges()` 尚未接入事件驱动
- 还没有真正持续维护“当前播放窗口”
- seek 抢占仍偏弱
- 旧窗口降级与新窗口提权更多是静态反应，不是完整调度器

### 5.3 结论

当前代码更接近：

- “有优化雏形”

而不是：

- “完整请求驱动调度器”

这和 `todolist.md` 中“滑动窗口策略未实现、Seek 抢占未实现、SubscribePieceStateChanges() 未实现”的判断是相互印证的。

## 6. 优化方案对比

这里给出三套可执行方案。

### 方案 A：先修调度时机

#### 目标

把 Range/seek 提权从“响应结束后”前移到“请求到来时”。

第一阶段只做最小闭环：在 `ServeContent(...)` 前复用现有 `applyRangePriority(...)`，按当前 Range 提前触发热点窗口提权。

#### 预期收益

- 改善首播速度
- 改善 seek 恢复速度
- 降低播放器一次 seek 后的卡顿时间

#### 优点

- 收益直接
- 改动面相对集中
- 不需要先重构整体架构
- 可以保留现有响应后观测与动态 readahead 链路

#### 风险

- 需要小心处理 Range 解析、请求失败、无 Range 请求等边界情况
- 如果请求开始阶段污染了响应后统计语义，会让 seek 判定失真
- 若一次性混入窗口回收、事件订阅、deadline 调度，会放大回归面

#### 结论

这是当前最应该优先落地的方案。

---

#### 综合评价

推荐优先级：**最高**

---

### 方案 B：治理高频持久化

#### 目标

把 session 的“恢复性数据”和“高频观测数据”分层，不再让每次 metrics 更新都整份落盘。

#### 可选方向

1. 只把会话创建、删除、关键生命周期变化持久化
2. 高频观测字段只保存在内存
3. 采用定时批量刷盘
4. 将观测数据与恢复数据拆文件

#### 预期收益

- 降低磁盘写放大
- 降低 UI/观测链路对下载链路的干扰
- 提高多会话场景稳定性

#### 风险

- 会引入“程序异常退出后部分指标不保留”的权衡

#### 综合评价

推荐优先级：**高**

---

### 方案 C：补齐设置生效语义与网络侧优化

#### 目标

解决“设置看似已更新，但未完整作用到底层 client”的问题，并继续增强资源发现能力。

#### 可选方向

1. 明确哪些配置需要重启才能生效
2. 允许在设置变更后安全重建 torrent client
3. 补充慢 peer 治理、deadline、请求策略优化
4. 继续增强 tracker / DHT 诊断与可视化

#### 预期收益

- 调参结果更可信
- 网络侧问题更容易定位
- 对低热资源和弱资源更友好

#### 风险

- 改动面更大
- 更容易触及运行期资源管理与生命周期复杂性

#### 综合评价

推荐优先级：**中**

---

## 7. 推荐执行顺序

建议不要平均推进，而是按收益排序逐步推进：

1. 先做 Range 提权时机前移
2. 再做会话观测数据与恢复数据分层，降低刷盘频率
3. 再补齐滑动窗口与 seek 抢占
4. 再补 `SubscribePieceStateChanges()` 事件驱动
5. 最后处理设置热生效、慢节点治理和更深入的网络层优化

这个顺序的原因是：

- 第 1 步最直接改善体感
- 第 2 步最直接降低系统抖动
- 第 3、4 步是把调度器从“雏形”升级成“完整能力”
- 第 5 步更适合在前面证据收集充分后推进

---

## 8. 验证与观测计划

后续做优化时，不建议只盯住“总下载速度”。这个项目核心不是纯下载器，而是边下边播，因此需要同时看四类指标：

### 8.1 关键指标

- metadata 获取耗时
- 第一帧时间
- seek 恢复时间
- 平均 Range 响应耗时
- buffer hit ratio
- useful bytes 增量
- ActivePeers / TotalPeers

### 8.2 测试场景

至少覆盖三类资源：

1. 高热公开资源
2. 低热公开资源
3. 私有资源或 tracker 依赖型资源

除此之外，至少再覆盖两类媒体形态：

4. 头部元数据友好的资源
5. 更依赖尾部索引或更容易触发尾部探测的资源

### 8.3 对比方式

每次优化都应该做前后对比：

- 同一资源
- 同一播放器
- 同一网络环境
- 同一设置
- 同一磁盘环境

否则无法说明优化是否真实有效。

### 8.4 建议测试矩阵

如果后续要把下载速度和边下边播优化做成可复用结论，建议至少形成以下最小矩阵：

1. **资源维度**
   - 高热 / 低热 / 私有
2. **播放器维度**
   - 偏单段顺序读取
   - 偏多次探测与频繁 seek
3. **容器维度**
   - 头部元数据友好
   - 尾部索引更敏感
4. **环境维度**
   - SSD
   - HDD
   - 单会话
   - 多会话

如果没有这个矩阵，就很容易把“只对某一种播放器 + 某一种资源有效”的结果误判成通用优化。

## 9. 苏格拉底式质询与结论校验

### 第一轮质询

#### 对手 A

也许真正影响下载速度的是资源本身 peer 太少，不是你的代码。

#### 回应

成立，但不完整。

资源质量决定“上限”，调度时机决定“你离上限有多远”。当前代码中提权发生在响应结束后，这属于内部可控问题，应该优先修。

#### 当前置信

高。因为这是明确的调用时序问题。

### 第二轮质询

#### 对手 B

就算提权时机有问题，Reader 自己已经有 `SetResponsive()` 和 `SetReadahead()` 了，未必值得再动。

#### 回应

Reader 自带优先级没错，但当前项目已经显式加入了 `applyRangePriority(...)`、seek gap、boost window 等扩展逻辑，说明仅依赖 Reader 默认行为并不满足目标。既然已经选择在库能力之上再做调度，就应该把关键时机做对。

#### 当前置信

高。因为现有代码和 `todolist` 都说明项目目标早已超出“只用默认 Reader 即可”。

### 第三轮质询

#### 对手 C

刷盘热点也许只是理论问题，真实机器上未必影响下载。

#### 回应

对单会话、SSD、高热资源场景，影响可能不明显；但在多会话、频繁 Range、Windows 磁盘忙时，持续整文件落盘会放大系统抖动。这属于典型的系统性隐患，不应等到症状明显才处理。

#### 当前置信

中高。逻辑成立，但具体影响幅度仍需实测量化。

### 第四轮质询

#### 对手 D

也许协议细节和容器布局只是边缘问题，真正决定播放体验的还是 peer 数量和下载吞吐。

#### 回应

这只在“资源供给极充足，且播放器请求模式比较朴素”的场景更接近成立。对于边下边播，播放器究竟请求哪个区间、是否先探尾部、是否发多段 `Range`，会直接决定调度器是否把有限带宽用在最急需的数据上。供给侧决定上限，协议和容器语义决定“你有没有把上限用对地方”。

#### 当前置信

中高。方向判断可靠，但不同播放器和容器的影响幅度仍需矩阵化实测。

## 10. 最终结论

### 10.1 总结判断

当前项目的下载和边下边播速度**还能继续提升**，而且提升空间并不小。

最关键的问题不是“完全缺少优化能力”，而是：

1. 请求驱动调度的关键时机还不对
2. 高频观测更新带来了不必要的整文件落盘
3. 设置更新与底层 client 生效之间存在语义断层
4. 协议边界与容器差异尚未被系统化纳入调度和观测模型

### 10.2 推荐优先级

优先顺序建议固定为：

1. Range 提权时机前移
2. 会话刷盘治理
3. 滑动窗口 + seek 抢占补全
4. PieceStateChanges 事件驱动
5. 协议边界建模与测试矩阵落地
6. 设置热生效与网络侧治理

### 10.3 当前置信等级

**高**

支撑依据：

- `todolist.md` 与现有代码状态一致
- 核心问题可由明确代码路径证明
- 问题类型属于典型流式调度与状态持久化协同问题

---

## 11. 残余不确定性

以下部分仍需要真实资源与播放器实测才能完全量化：

1. 不同播放器的 Range 模式差异，对 seek 恢复的影响幅度
2. 低热资源下，调度优化与 peer 质量之间的真实收益占比
3. `sessions.json` 高频刷盘在不同磁盘环境下的影响程度
4. 设置热生效是否值得做成运行期重建 client，而不是要求重启生效
5. 多段 `Range`、尾部 `suffix range`、`If-Range` 在真实播放器中的出现频率
6. 容器元数据位置对首播/seek 的影响在当前资源样本里占比多大

这些不确定性不会推翻本文主结论，但会影响后续优化排期中的收益预估。
