# BTDown_MA 流式播放与下载优化执行路线图

## 1. 文档目的

本文档是在以下两份已有文档基础上的进一步收敛：

- `docs/download-speed-analysis-plan.md`
- `docs/open-source-streaming-performance-research.md`

目标不是继续做“问题分析”，而是把结论映射成当前项目可直接执行的阶段计划，回答四个问题：

1. 接下来应该先做什么。
2. 每个阶段的完成标准是什么。
3. 哪些任务必须前置，哪些可以后置。
4. 如何判断当前阶段是否可以安全进入下一阶段。

本文档只做执行路线规划，不包含业务代码修改。

---

## 2. 当前项目位置判断

结合 `todolist.md`、现有实现和前两份分析文档，当前项目更准确的位置是：

- 已经完成“可播放的流式 MVP”
- 还没有完成“可稳定播放、可快速 seek、可高质量诊断”的流式系统

换句话说，当前状态不是从 0 到 1，而是：

- 从 `能播` 往 `播得稳、播得快、可优化` 演进

因此路线图不建议再走“大而全重构”，而是按收益和风险拆成几个连续阶段逐步推进。

---

## 3. 总体策略

后续阶段安排遵循三个原则：

### 3.1 先修热路径，再修外围能力

优先修复真正影响首播和 seek 的热路径：

- Range 请求调度时机
- 时间关键片段优先级
- 关键窗口维护

而不是优先做：

- 更复杂的 UI
- 更多设置项
- 更完整的播放器联动

### 3.2 先建立证据链，再扩大功能面

如果没有稳定的观测指标，就很难判断：

- 优化是否真的有效
- 卡顿到底发生在网络、调度、磁盘还是播放器行为层

因此每个阶段都必须配套观测和验收指标。

### 3.3 先形成“稳定策略”，再开放“高级参数”

当前项目已有不少底层参数，但后续不建议继续优先加更多参数。

应先把系统做成少数几种可靠策略，再考虑是否开放更细的调优入口。

---

## 4. 路线图总览

建议分为 6 个 Phase 推进：

1. Phase A：调度时机纠正
2. Phase B：事件驱动调度闭环
3. Phase C：播放窗口与 seek 抢占升级
4. Phase D：缓存与持久化治理
5. Phase E：网络与 peer 调度增强
6. Phase F：产品化与策略封装

这 6 个 Phase 的设计顺序是：

- 先解决当前最直接影响体验的问题
- 再做结构性优化
- 最后收敛成可交付产品能力

---

## 5. Phase A：调度时机纠正

### 5.1 阶段目标

把当前流式调度从“请求后补偿”纠正为“请求到来时主动准备”。

这是当前整个路线图的第一优先级。

本阶段只追求一个最小闭环：`ServeContent(...)` 开始响应前，已经根据当前 `Range` 把热点 piece 窗口提到更高优先级。

### 5.2 背景问题

当前代码路径里，Range 相关统计和 priority 调整发生在请求结束后。

这意味着：

- 对下一次请求可能有帮助
- 对当前首播请求和当前 seek 请求帮助有限

### 5.3 本阶段任务

1. 将 Range 请求处理拆成“请求开始”和“请求结束”两个时点
2. 在请求开始时完成：
   - Range 解析
   - 轻量 seek 粗判定
   - 当前热点窗口定位
   - 提前提升关键 piece 优先级
3. 在请求结束时保留：
   - 耗时记录
   - 首帧/seek 恢复指标回填
   - 命中率统计
   - 动态 readahead 调整
4. 第一阶段复用现有 `applyRangePriority(...)`，不在本阶段重写完整滑动窗口调度器
5. 明确把以下工作后置到后续阶段：
   - `SubscribePieceStateChanges()` 事件驱动闭环
   - 完整窗口升降级与旧窗口回收
   - time-critical / deadline 驱动调度
   - 高频观测与持久化治理

### 5.4 验收标准

满足以下条件才算进入下一阶段：

1. `ServeContent(...)` 前已经触发请求开始提权
2. 原有响应后观测链路仍然完整，首帧/seek/命中率指标没有丢失
3. 首帧时间在相同资源下有稳定下降趋势
4. seek 恢复时间相较当前基线有明显改善
5. 没有因为前移调度导致新的死锁、阻塞或异常状态错判

### 5.5 风险与注意事项

- 不要在请求开始时做过重操作，避免反而拉长主请求延迟
- 对无 `Range` 请求、非法 `Range` 请求、`HEAD` 请求要有边界处理
- 本阶段不要试图一次做完整 deadline 调度，先把“时机纠正”做好

---

## 6. Phase B：事件驱动调度闭环

### 6.1 阶段目标

把当前依赖轮询和事后统计的调度方式，升级成更实时的事件驱动模式。

### 6.2 背景问题

当前项目虽然已经有：

- 每秒 metrics 刷新
- Range 请求观测

但还没有把 `SubscribePieceStateChanges()` 真正接入。

这意味着：

- 调度器对 piece 状态变化的感知仍不够及时
- seek 恢复过程更多依赖下一次观测，而不是即时反馈

### 6.3 本阶段任务

1. 接入 `SubscribePieceStateChanges()`
2. 建立 piece 状态变化到 session/调度器的事件通路
3. 明确哪些事件会触发：
   - 窗口重算
   - buffer 命中判断
   - seek 恢复完成
   - 关键块等待超时
4. 将“播放状态推进”从纯轮询判断升级为“轮询 + 事件”混合模式

### 6.4 验收标准

1. 关键 piece 完成后，调度器和观测层能及时感知
2. seek 恢复完成不再完全依赖后置日志推断
3. 在高频播放请求下，不出现事件风暴导致的 UI 或仓储压力
4. 指标与状态更新的实时性优于当前纯轮询模式

### 6.5 风险与注意事项

- 事件接入后要严格节流，避免把高频 piece 变化直接倾倒给 UI
- 事件通路要和热路径解耦，不能让观测链路反向拖慢下载

---

## 7. Phase C：播放窗口与 seek 抢占升级

### 7.1 阶段目标

将现有的 boost window 雏形升级成真正可工作的播放窗口调度器。

### 7.2 本阶段任务

1. 明确三类区域：
   - 当前播放窗口
   - 预读窗口
   - 窗口外普通下载区
2. 设计窗口移动规则：
   - 顺序播放时平滑推进
   - seek 时快速重置
3. 设计旧窗口降级策略：
   - 什么时候立即降级
   - 什么时候保留一小段热区
4. 为 seek 增加更强的抢占逻辑
5. 评估是否引入更细粒度的“时间关键片段”概念，而不是只按静态窗口提权

### 7.3 验收标准

1. 连续播放期间窗口移动稳定，不频繁抖动
2. seek 到远位置后，优先级可以快速切换到新区域
3. 首次 seek 恢复耗时明显小于当前版本
4. 窗口外下载不会被完全饿死

### 7.4 风险与注意事项

- 如果旧窗口降级太快，短时间回退 seek 可能体验变差
- 如果旧窗口保留太久，新 seek 抢占能力会变弱
- 本阶段要特别关注“窗口内体验”和“全局 swarm 利用率”的平衡

---

## 8. Phase D：缓存与持久化治理

### 8.1 阶段目标

降低系统内部不必要的 I/O 干扰，让下载热路径更干净。

### 8.2 背景问题

当前已识别到的结构性问题包括：

1. session 高频更新时存在整文件落盘风险
2. cache/buffer 还没有被作为系统级设计对象对待

### 8.3 本阶段任务

1. 拆分“恢复性数据”和“高频观测数据”
2. 降低 `sessions.json` 高频重写
3. 评估以下几种策略：
   - 生命周期事件即时持久化
   - 高频指标内存化
   - 定时批量刷盘
   - 观测与恢复分文件
4. 把缓存和缓冲区纳入统一设计：
   - reader readahead
   - 本地缓存热区
   - seek 后短期热区保留

### 8.4 验收标准

1. 高频播放和多会话下磁盘抖动明显降低
2. session 恢复能力不退化
3. 观测数据与恢复数据的职责边界清晰
4. cache/buffer 策略开始具备统一语义

### 8.5 风险与注意事项

- 不要为了减少刷盘而丢掉必要的崩溃恢复能力
- 缓存设计不能只追求大，要围绕播放器实际请求模式

---

## 9. Phase E：网络与 peer 调度增强

### 9.1 阶段目标

从“能调 piece”提升到“能调 peer 和交付能力”。

### 9.2 背景问题

当前可观测层已经能看到：

- `ActivePeers`
- `TotalPeers`
- useful bytes
- DHT/tracker 基础诊断

但这还不足以支撑更高质量的时间关键调度。

### 9.3 本阶段任务

1. 引入更细的 peer 质量判断
2. 识别慢 peer 对关键块交付的影响
3. 为关键块设计超时补救策略
4. 区分：
   - 首播期调度
   - seek 恢复期调度
   - 稳态播放期调度
5. 继续完善 tracker / DHT / 入站连接的诊断解释层

### 9.4 验收标准

1. 弱资源场景下 seek 恢复更稳定
2. 活跃 peer 数量不再是唯一判断依据
3. 能从观测面上回答“为什么这次卡住”
4. 关键块等待超时后有明确补救路径

### 9.5 风险与注意事项

- 不要在证据不足时过度设计复杂 peer 排名模型
- 要先用观测证明慢 peer 问题存在，再引入更重策略

---

## 10. Phase F：产品化与策略封装

### 10.1 阶段目标

把前面阶段形成的底层能力封装成稳定、可解释、可交付的产品策略。

### 10.2 本阶段任务

1. 将零散参数逐步收敛为高层模式，例如：
   - 快速首播
   - 平衡模式
   - 稳播模式
2. 明确哪些设置：
   - 运行期立即生效
   - 需要重启生效
   - 仅用于高级调试
3. 为 observability 界面补充更直接的解释文案
4. 输出面向用户和开发者的调优文档
5. 整理与播放器联动相关的可用性策略

### 10.3 验收标准

1. 常见用户不需要理解底层 BT 参数也能选择合适模式
2. 设置语义明确，不再出现“改了但不确定是否生效”的体验
3. 诊断结果可直接指导问题排查
4. 项目从“工程试验品”更接近“可维护产品”

### 10.4 风险与注意事项

- 在底层能力没稳定前，不要过早包装太多 UI 策略
- 产品化不是“隐藏问题”，而是“建立稳定抽象”

---

## 11. 每个阶段的依赖关系

### 11.1 强依赖

- Phase B 依赖 Phase A
- Phase C 依赖 Phase A，并强烈受益于 Phase B
- Phase E 依赖 Phase C 的窗口与 seek 基础能力
- Phase F 依赖前面阶段产出的稳定能力

### 11.2 可并行但不建议过早推进的部分

- Phase D 中的持久化治理可以部分独立推进
- 但缓存策略设计最好在 Phase C 之后再定型

### 11.3 不建议提前推进的内容

以下内容不建议优先于 Phase A/B/C：

- 内嵌播放器
- 更复杂的前端展示
- 更多裸参数暴露
- 大范围 UI 重构

---

## 12. 指标体系与验收门槛

为了避免“做了很多优化，但不知道有没有变好”，建议路线图全过程统一追踪以下指标。

### 12.1 核心指标

1. metadata 获取耗时
2. 第一帧时间
3. seek 恢复时间
4. 平均 Range 响应耗时
5. buffer hit ratio
6. useful bytes 增量
7. ActivePeers / TotalPeers

### 12.2 建议门槛

进入下一阶段前，至少应满足：

1. 当前阶段的关键体验指标有稳定改善
2. 没有引入明显新的退化
3. 问题定位能力比进入当前阶段前更强
4. 关键指标可以重复采样，而不是偶然结果

### 12.3 测试场景

每个阶段建议至少覆盖：

1. 高热公开资源
2. 低热公开资源
3. tracker 依赖更强的资源
4. 不同播放器的基本 Range 行为

---

## 13. 执行节奏建议

建议采用如下推进节奏：

1. 先完成 Phase A
2. 用最小真实资源集做回归
3. 再推进 Phase B
4. Phase B 稳定后，再进入 Phase C
5. Phase C 稳定后，开始推进 Phase D 和 Phase E
6. 当前三层能力稳定后，再做 Phase F 产品化收敛

这个节奏的原因是：

- A/B/C 决定核心播放体验
- D/E 决定系统稳定性和上限
- F 决定可交付性

---

## 14. 推荐的近期落地顺序

如果只看接下来最值得投入的短期行动，建议顺序固定为：

1. 先做 Phase A：纠正调度时机
2. 再做 Phase B：接事件驱动
3. 再做 Phase C：补窗口与 seek 抢占
4. 紧接着做 Phase D：治理刷盘与缓存
5. 最后进入 Phase E/F

这是当前最稳妥、也最符合收益最大化的推进顺序。

---

## 15. 最终结论

对当前 `BTDown_MA` 来说，最合理的演进路径不是“大重写”，而是：

1. 先把流式调度从“事后补偿”修成“请求时主动调度”
2. 再把轮询式调度补成事件驱动调度
3. 再把窗口提权补成更完整的播放窗口/seek 抢占体系
4. 然后清理缓存和持久化等系统性干扰项
5. 最后把能力封装成真正可交付的播放策略

如果这条路线执行到位，项目会从当前的：

- `能播的流式 MVP`

逐步升级为：

- `可稳定播放、seek 更快、问题更容易定位的桌面流式系统`
